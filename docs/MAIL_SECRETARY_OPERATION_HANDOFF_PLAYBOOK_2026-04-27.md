# 메일 비서 시스템 실행 플레이북 (착수/테스트/업로드/수동작업)

## 1) 개발 착수: 효율적 업무분장

## 1.1 Squad 분장표 (RACI 간소화)
| 업무영역 | 담당 Squad | 핵심 산출물 | 선행조건 |
|---|---|---|---|
| 플랫폼/배포 | Squad A (Platform) | Cloud Run 배포, Scheduler OIDC, IAM 최소권한 | 없음 |
| 메일연동 | Squad B (Mail) | Gmail 조회/회신, thread 유지, 첨부 발송 | Secret 로더 |
| 보안정책 | Squad C (Security) | 발신자 allowlist, PII 탐지, 마스킹, 고위험 차단 | 정책 룰셋 |
| AI오케스트레이션 | Squad D (AI) | 분류기, 라우터, 모델 fallback, timeout/retry | 보안필터 인터페이스 |
| 출력/검증 | Squad E (Output+QA) | 1,000자 본문 빌더, PDF 생성, 테스트 자동화 | AI 응답 스키마 |

## 1.2 착수 순서 (병렬 최적화)
- Day 1: 공통 인터페이스 고정(API/DTO/Error/Audit Schema)
- Day 2~4: Squad A~E 병렬 개발
- Day 5~6: 통합(E2E) + 결함 수정
- Day 7: 스테이징 배포 점검
- Day 8: 파일럿 테스트 시작

## 1.3 일일 운영 규칙
- 인터페이스 변경은 아키텍트 승인 없이는 금지
- 오전 15분 스탠드업에서 blocker 공유
- 저녁 통합 빌드 1회 필수

## 2) 개발 완료 후 테스트 실행 절차

## 2.1 파일럿 테스트 4회 (비민감 샘플)
1. Pilot-1: 허용 발신자 + 일반 요약 요청
2. Pilot-2: 미허용 발신자 요청(SKIPPED 확인)
3. Pilot-3: 코드 포함 요청(PDF 강제 확인)
4. Pilot-4: 인증정보 포함 요청(외부 호출 차단 확인)

### 파일럿 통과 기준
- 4회 모두 PASS
- 보안 위반 0건
- 본문 1,000자 제한 위반 0건

## 2.2 전수 테스트 4회 (회귀 + 장애)
1. Full-1: TC-01~03
2. Full-2: TC-04~06
3. Full-3: TC-07~08
4. Full-4: TC-09~10 + 장애복구 시나리오

### 전수 통과 기준
- 4회 모두 PASS
- 재현 가능한 결함 0건
- dedup 정확도 100%

## 3) 특이사항 없을 시 GitHub 업로드 절차

1. 기능 브랜치 최신화 및 최종 커밋 정리
2. 테스트 리포트(파일럿4 + 전수4) 첨부
3. PR 생성 후 리뷰어 지정
4. 보안/운영 승인 완료
5. main 머지 및 버전 태깅
6. 릴리즈 노트/변경이력 반영

---

## 4) 사용자가 수동으로 해야 하는 과정 설명서

1. GitHub에서 최종 PR 열기
2. 아래 체크리스트를 직접 확인
   - [ ] 파일럿 테스트 4회 PASS
   - [ ] 전수 테스트 4회 PASS
   - [ ] 고위험 민감정보 차단 PASS
   - [ ] Secret 노출 없음
   - [ ] Scheduler/Run IAM 최소권한 확인
3. 조직 승인자(보안/운영) 승인 버튼 클릭
4. Merge 방식 선택(Squash/Rebase 정책 준수)
5. 릴리즈 태그(예: v1.0.0) 생성
6. Cloud Scheduler Job 활성화(운영 시작 시점)
7. 첫 1회 배치 실행 후 로그 모니터링 결과 승인
